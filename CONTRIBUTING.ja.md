# xai-cli へのコントリビューション

xai-cli へのご関心をありがとうございます！このドキュメントでは、プロジェクトへのコントリビューションに関するガイドラインと手順を説明します。

## ようこそ

xai-cli は、xAI API (Grok) を通じて X (Twitter) の投稿を検索・分析するための CLI ツールです。バグ修正・機能追加・ドキュメント改善など、あらゆる形でのコントリビューションを歓迎します。

## コントリビューションの方法

以下の方法でコントリビューションできます。

- **バグ報告** — 問題を見つけたら教えてください！
- **機能提案** — アイデアがあればぜひ共有してください
- **ドキュメント改善** — ドキュメントをより分かりやすくする手助けをしてください
- **バグ修正の提出** — 問題を修正して安定性の向上に貢献してください
- **新機能の追加** — xai-cli の機能を拡張してください（事前に議論をお願いします）

## 始める前に

1. **既存の Issue / PR を確認**して重複を避けてください
2. **新機能の場合**、まず Issue を作成して提案を議論してください
3. **[テストガイド](docs/TESTING.md)** を読んで、テスト方針を理解してください
4. **[行動規範](CODE_OF_CONDUCT.ja.md)** を確認・理解してください

## 開発環境のセットアップ

### 前提条件

- Node.js 18+ / npm 9+
- xAI API キー（[API キーの取得方法](https://x.ai/api)）

### セットアップ手順

```bash
# 1. リポジトリをフォークしてクローン
git clone https://github.com/YOUR_USERNAME/xai-cli.git
cd xai-cli

# 2. 依存パッケージをインストール
npm install

# 3. 環境変数を設定
export XAI_API_KEY=your_api_key

# 4. テストを実行してセットアップを確認
npm test

# 5. プロジェクトをビルド
npm run build

# 6. CLI をローカルでテスト
npm link
xai --version
```

## ローカルスキル互換性

複数の X 関連ローカルスキルは、X / xAI API への唯一の承認済み入口として `xai` コマンドに依存しています。スキル側で必要になった機能を `curl`、直接 HTTP、一時スクリプトで回避せず、必要なコマンド・オプション・JSON 項目・trace 機能を `xai-cli` に追加してください。

`ask`、`search`、`tweet`、`thread`、`user`、出力モード、trace flags に触れる変更では後方互換を維持し、破壊的変更が避けられない場合は移行手順を明記してください。CLI 挙動を変える場合は README の smoke check を実行します。

## コーディング規約

### TypeScript スタイル

- **strict モード**を使用する（設定済み）
- `let` より `const` を優先し、`var` は使わない
- 意味のある変数名を使う（`h` ではなく `handle`）
- `any` 型を避ける — 必要な場合は `unknown` を使う
- 関数のパラメーターと戻り値には型を付ける

### コードの構成

- 関数を小さく単一の責務に保つ
- 複雑なロジックは別の関数に切り出す
- ロジックが自明でない場合のみコメントを追加する
- コードベースの既存パターンに従う

### コミットメッセージの規約

形式: `<type>: <subject>`

**タイプ一覧:**

- `feat:` 新機能
- `fix:` バグ修正
- `test:` テストの追加・変更
- `docs:` ドキュメントの変更
- `refactor:` コードのリファクタリング（機能変更なし）
- `chore:` メンテナンス作業（依存更新・ツール設定等）

**例:**

```text
feat: add support for image understanding in search
fix: correct handle parsing for twitter.com URLs
test: add validation tests for ask command
docs: update README with new --exclude flag
refactor: extract URL parsing logic to separate module
chore: update dependencies to latest versions
```

## テスト要件

**すべてのコードコントリビューションにはテストを含める必要があります。**

### テストの種類

1. **ユニットテスト** — 個々の関数を独立してテスト
2. **バリデーションテスト** — CLI 入力バリデーションのロジックをテスト
3. **出力テスト** — CLI の出力フォーマット（テキストおよび JSON）をテスト
4. **エラーハンドリングテスト** — エラーシナリオと終了コードをテスト

### テストの実行

```bash
# 全テストを実行
npm test

# 開発中にウォッチモードで実行
npm run test:watch

# 特定のテストファイルを実行
npm test src/lib/__tests__/client.test.ts
```

### テスト作成のガイドライン

- **Arrange / Act / Assert** パターンに従う
- できる限り 1 テストにつき 1 アサーション
- 説明的なテスト名を使う: `it("should strip @ prefix from handle", ...)`
- 外部依存（`fetch`、`process.env`）はモック化する
- 包括的なテストガイドは **[docs/TESTING.md](docs/TESTING.md)** を参照

### テストカバレッジの期待値

- **新機能**: 新しいコードは 100% カバレッジ
- **バグ修正**: バグを再現するリグレッションテストを追加
- **リファクタリング**: 既存のカバレッジを維持または改善

## 変更の提出

### プルリクエストのプロセス

#### 1. ブランチを作成する

```bash
git checkout -b feat/your-feature-name
# または
git checkout -b fix/your-bug-fix
```

#### 2. 変更を加える

- コードを書く
- テストを追加する
- 必要に応じてドキュメントを更新する

#### 3. 品質を確認する

```bash
npm test          # すべてのテストがパスすること
npm run build     # ビルドが成功すること
```

#### 4. 変更をコミットする

```bash
git add .
git commit -m "feat: add your feature description"
```

#### 5. プッシュして PR を作成する

```bash
git push origin feat/your-feature-name
# その後、GitHub UI で PR を作成する
```

#### 6. PR テンプレートを記入する

- 何を、なぜ変更したかを説明する
- `Closes #123` で関連 Issue をリンクする
- テストの証拠を提供する
- テンプレートのチェックボックスをすべて確認する

### PR 提出前のチェックリスト

提出前に以下を確認してください。

- すべてのテストがパスしている（`npm test`）
- ビルドが成功している（`npm run build`）
- プロジェクトのスタイルに従っている
- コミットメッセージが規約に従っている
- 新機能にはテストが追加されている
- ドキュメントが更新されている（該当する場合）
- PR テンプレートが完全に記入されている

### 期待できること

- **初回レビュー**: 2〜3 営業日以内
- **フィードバック**: メンテナーからの変更リクエストやコメント
- **承認とマージ**: すべての要件が満たされた後

## コードレビュープロセス

### コントリビューターへ

- フィードバックや質問には**迅速に対応する**
- フィードバックが不明瞭な場合は**明確化を求める**
- 同じブランチに更新を**プッシュする**（PR は自動更新される）
- プロセスを通じて**忍耐強く、敬意を持って**臨む

### レビューの観点

レビュアーは以下を確認します。

- **機能性** — 意図した通りに動作するか？
- **テスト** — テストは網羅的でパスしているか？
- **コード品質** — 読みやすく、保守しやすいか？
- **ドキュメント** — 明確で最新の状態か？
- **パフォーマンス** — 明らかなパフォーマンス問題はないか？
- **セキュリティ** — 潜在的な脆弱性はないか？
